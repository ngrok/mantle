A persistent, decision-bearing bar that floats near the bottom edge of the viewport. It surfaces
pending state — primarily a form's unsaved ("dirty") changes — and stays until the user resolves
it: save or discard.

Toast announces something that already happened and leaves on its own; **Sandbar surfaces
something pending and stays until the user resolves it**. If the message needs no decision, reach
for [Toast](/components/feedback/toast) or [Alert](/components/feedback/alert) instead.

The name: a sandbar is a bar that blocks navigation — exactly what this component does while
changes are pending. Wire your router's navigation guard to `shake()` on the `SandbarHandle`, and
blocked attempts wiggle the panel and announce assertively to screen readers.

```tsx
import { Button } from "@ngrok/mantle/button";
import { Checkbox } from "@ngrok/mantle/checkbox";
import { Field } from "@ngrok/mantle/field";
import { Input } from "@ngrok/mantle/input";
import { Label } from "@ngrok/mantle/label";
import { Sandbar, type SandbarHandle } from "@ngrok/mantle/sandbar";
import { TextArea } from "@ngrok/mantle/text-area";
import { makeToast, Toast, Toaster } from "@ngrok/mantle/toast";
import { useRef, useState } from "react";

type Profile = {
	name: string;
	description: string;
};

const initialProfile: Profile = {
	name: "my-agent-endpoint",
	description: "Forwards traffic to the local dev server.",
};

export function EndpointSettings() {
	const [saved, setSaved] = useState<Profile>(initialProfile);
	const [draft, setDraft] = useState<Profile>(initialProfile);
	const [isPending, setIsPending] = useState(false);
	const [error, setError] = useState<string | null>(null);
	const [failNextSave, setFailNextSave] = useState(false);
	const sandbarHandle = useRef<SandbarHandle>(null);

	const isDirty = draft.name !== saved.name || draft.description !== saved.description;

	const save = () => {
		setIsPending(true);
		setError(null);
		// stand-in for the real request; fails on demand so the error row is visible
		window.setTimeout(() => {
			setIsPending(false);
			if (failNextSave) {
				setFailNextSave(false);
				setError("Something went wrong while saving. Try again.");
				return;
			}
			setSaved(draft);
			// the bar exits silently — announce the save's resolution yourself
			makeToast(
				<Toast.Root intent="success">
					<Toast.Icon />
					<Toast.Message>Changes saved</Toast.Message>
				</Toast.Root>,
			);
		}, 800);
	};

	const discard = () => {
		setDraft(saved);
		setError(null);
	};

	return (
		<form>
			<Field.Item name="endpoint-name">
				<Field.Label>Endpoint name</Field.Label>
				<Field.Control>
					<Input
						onChange={(event) => setDraft({ ...draft, name: event.target.value })}
						value={draft.name}
					/>
				</Field.Control>
			</Field.Item>
			<Field.Item name="endpoint-description">
				<Field.Label>Description</Field.Label>
				<Field.Control>
					<TextArea
						onChange={(event) => setDraft({ ...draft, description: event.target.value })}
						value={draft.description}
					/>
				</Field.Control>
			</Field.Item>
			<Label className="flex items-center gap-2" htmlFor="fail-next-save">
				<Checkbox
					checked={failNextSave}
					id="fail-next-save"
					onChange={(event) => setFailNextSave(event.target.checked)}
				/>
				Fail the next save (shows the error row)
			</Label>
			<Button
				appearance="link"
				intent="neutral"
				type="button"
				onClick={() => {
					// stand-in for a router navigation guard: while dirty, block
					// the navigation and shake the bar
					if (isDirty) {
						sandbarHandle.current?.shake();
					}
				}}
			>
				Continue to dashboard →
			</Button>

			<Sandbar.Root open={isDirty} handleRef={sandbarHandle}>
				<Sandbar.Message>You have unsaved changes</Sandbar.Message>
				<Sandbar.Actions>
					<Sandbar.DiscardButton disabled={isPending} onClick={discard}>
						Discard
					</Sandbar.DiscardButton>
					<Sandbar.SaveButton isLoading={isPending} onClick={save}>
						{isPending ? "Saving…" : "Save changes"}
					</Sandbar.SaveButton>
				</Sandbar.Actions>
				{error != null && <Sandbar.Error>{error}</Sandbar.Error>}
			</Sandbar.Root>
			<Toaster />
		</form>
	);
}
```

## Composition

Compose the parts of a `Sandbar` together to build your own:

```text showLineNumbers=false
Sandbar.Root
├── Sandbar.Message
├── Sandbar.Actions
│   ├── Sandbar.DiscardButton
│   └── Sandbar.SaveButton
└── Sandbar.Error
```

`Sandbar.Root` must stay mounted and be toggled with the controlled `open` prop — **never
conditionally mount it** (`{isDirty && <Sandbar.Root …>}`). The screen-reader announcement only
works when the internal live regions exist in the tree before the message appears, and the exit
animation needs the panel alive to play. A Root that mounts with `open` already true does not
announce (mounting is not a live-region change).

## Pending decisions beyond saving

Batch-saving a form is the flagship use, but any pending decision that must be resolved before
moving on fits: compose your own `Button`s inside `Sandbar.Actions` and pass a custom
blocked-navigation announcement to `shake()` so screen-reader users hear the actions that actually
exist on screen. The panel's accessible name follows `Sandbar.Message` automatically.

<CodeExample.Root>
	<CodeExample.PreviewFrame example="sandbar-pending-publish" title="Sandbar pending publish demo" />

```tsx
import { Button } from "@ngrok/mantle/button";
import { Sandbar, type SandbarHandle } from "@ngrok/mantle/sandbar";
import { useRef, useState } from "react";

const draftPosts = [
	{ id: 1, title: "Introducing the canvas chart family" },
	{ id: 2, title: "Traffic policy recipes for rate limiting" },
	{ id: 3, title: "What's new in the dashboard" },
];

export function DraftPosts() {
	const [pending, setPending] = useState(draftPosts);
	const sandbarHandle = useRef<SandbarHandle>(null);

	return (
		<>
			<ul>
				{pending.map((draft) => (
					<li key={draft.id}>{draft.title}</li>
				))}
			</ul>
			{pending.length === 0 && <p>All drafts published.</p>}
			<Button
				appearance="link"
				intent="neutral"
				type="button"
				onClick={() => {
					if (pending.length > 0) {
						sandbarHandle.current?.shake({
							announcement: "Publish or discard your drafts before leaving.",
						});
					}
				}}
			>
				Continue to dashboard →
			</Button>

			<Sandbar.Root open={pending.length > 0} handleRef={sandbarHandle}>
				<Sandbar.Message>
					{pending.length} draft {pending.length === 1 ? "post" : "posts"} pending publish
				</Sandbar.Message>
				<Sandbar.Actions>
					<Button
						appearance="outlined"
						intent="neutral"
						type="button"
						onClick={() => setPending([])}
					>
						Discard drafts
					</Button>
					<Button appearance="filled" intent="neutral" type="button" onClick={() => setPending([])}>
						Publish all
					</Button>
				</Sandbar.Actions>
			</Sandbar.Root>
		</>
	);
}
```

## Polymorphism

`Sandbar.Message` and `Sandbar.Actions` accept an `asChild` prop. When `true`, the part renders
its single child instead of its default element (`<p>` and `<div>` respectively), forwarding all
class names, `data-*` attributes, and the ref onto that child. Reach for it when the message needs
a different semantic element (e.g. a `<span>` inside a larger sentence) or when the actions row
should be another framework primitive. `Sandbar.SaveButton` and `Sandbar.DiscardButton` inherit
`asChild` from [Button](/components/actions/button). `Sandbar.Root` deliberately does not support
`asChild`: it renders multiple nodes (positioning wrapper, two screen-reader announcers, and the
panel), so single-child polymorphism has no coherent meaning. Both framed demos above render real
`Sandbar` trees you can adapt; there is no separate polymorphism demo because the bar is a
viewport-fixed singleton that would overlay this page outside a preview frame.

## Theming

The panel is an [inverted theme island](/components/primitives/theme#inverted-theme-islands): it
renders in the opposite theme of the page — dark on light pages, light on dark pages, and the
same flip between the high-contrast pair. The inversion covers the entire subtree, so the blessed
buttons, `Sandbar.Error`, and any custom children you compose inside `Sandbar.Actions` all read as
the opposite theme without extra styling. The panel's own surface uses the inverted theme's
`bg-base` and `text-strong` tokens; style overrides passed via `className` resolve against the
inverted theme too.

## Blocking navigation

The bar renders the decision; your app owns the guard. Pair `open` with your router's blocker
(e.g. react-router's `useBlocker` plus a `beforeunload` listener while dirty) and call
`sandbarHandle.current?.shake()` when a navigation attempt is blocked — the hero example above
shows the wiring with a stand-in guard. Keep the guard out of the component: mantle has no router
dependency, and the split mirrors Shopify's SaveBar / `leaveConfirmation()` and UIKit's
`isModalInPresentation` / `presentationControllerDidAttemptToDismiss`.

## Accessibility

- **Announcements.** `Sandbar.Root` owns two persistent visually-hidden live regions. A polite
  `role="status"` announcer speaks the message when the bar opens and "Saving changes…" while
  `Sandbar.SaveButton` is loading. An assertive `role="alert"` announcer speaks blocked-navigation
  attempts (from `shake()`) and any `Sandbar.Error` content. This is why Root must stay mounted.
- **Announce success yourself.** The bar closes silently when changes resolve. Pair the save's
  resolution with a status announcement — a success [Toast](/components/feedback/toast) qualifies —
  or screen-reader users hear a save start and never finish.
- **Focus is never stolen.** The bar appears without taking focus. If focus is inside the bar when
  it closes (or a focused save button becomes disabled while loading), focus is parked on the
  panel and restored to the element focused before entering the bar once it closes — with guards:
  if you already moved focus elsewhere, the bar leaves it alone.
- **Escape is intentionally inert.** A one-keypress discard would destroy user data; resolving the
  pending state requires an explicit action.
- **Reduced motion.** Under `prefers-reduced-motion`, the enter/exit slides degrade to fades and
  `shake()` skips the wiggle — the assertive announcement still fires, so blocked navigation stays
  perceivable.
- **Zoom and small viewports.** The panel wraps its actions under the message and caps its height,
  so it stays usable at 320px-wide viewports (400% zoom).

## Focus not obscured

A fixed bar can cover the focused element behind it (WCAG 2.4.11). Two mitigations are built in:
the panel is a centered island rather than an edge-to-edge strip, and it floats above the bottom
edge. For long forms, additionally set `scroll-padding-bottom` (roughly the bar height plus its
offset, ~`6rem`) on the scrolling container while the bar is open, so keyboard-driven
scroll-into-view lands fields above the bar. Note that scroll padding only helps elements being
scrolled into view — an element already in view behind the bar is only partially obscured thanks
to the island shape, which satisfies the AA criterion.

## Placement

Render `Sandbar.Root` in place, inside or immediately after the form it saves — `position: fixed`
changes paint position, not DOM or tab order, so Tab from the last form field lands directly on
Discard/Save. Two caveats:

- An ancestor with `transform`, `filter`, or `will-change` becomes the containing block for
  `position: fixed` (common with page-transition wrappers). If you cannot render Root outside the
  transformed ancestor, portal it yourself with `createPortal(<Sandbar.Root …/>, document.body)` —
  and accept that this severs the form→actions tab-order adjacency.
- Only render one Sandbar per page: it is a singleton surface, like a page's primary save action.

## API Reference

### Sandbar.Root

The always-mounted shell: a private fixed-position wrapper, two persistent screen-reader
announcers, and the visible panel. `ref`, `className`, and all other props target the panel (a
`role="group"` element named by `Sandbar.Message` via `aria-labelledby`; a consumer `aria-label`
wins when passed). Does not support `asChild` — Root renders multiple nodes, so single-child
polymorphism has no coherent meaning.

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop         | Type                 | Default | Description                                                                                          |
| ------------ | -------------------- | ------- | ---------------------------------------------------------------------------------------------------- |
| `open`       | `boolean`            | —       | Whether the bar is showing. Controlled-only; drive it from your pending state (e.g. a dirty flag).   |
| `handleRef?` | `Ref<SandbarHandle>` | —       | Receives the imperative handle: `shake(options?: { announcement?: string })` for blocked navigation. |

### SandbarHandle

The imperative surface received through `handleRef`. `shake()` wiggles the panel (skipped under
`prefers-reduced-motion` — never rely on the motion alone) and always announces assertively. Pass
`announcement` to replace the default save-flavored wording for other intents.

| Method  | Signature                                       | Description                                                          |
| ------- | ----------------------------------------------- | -------------------------------------------------------------------- |
| `shake` | `(options?: { announcement?: string }) => void` | Wiggle the panel and announce that the attempted action was blocked. |

### Sandbar.Message

The visible pending-state text. A plain paragraph — the live-region announcement is owned by
Root's persistent announcer, and the panel's accessible name points here so the group name always
matches the visible text.

All props from [p](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/p#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Sandbar.Actions

The action-button row: a plain flex container (deliberately not `role="toolbar"`, which is
reserved for 3+ controls). Holds `Sandbar.DiscardButton` and `Sandbar.SaveButton`, or your own
composed `Button`s.

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                       |
| ---------- | --------- | ------- | ------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Render as a different element by passing a child. |

### Sandbar.SaveButton

The primary (save) action — a mantle [Button](/components/actions/button) defaulting to the system
primary style. While `isLoading`, Root announces "Saving changes…" politely and keeps focus from
being lost when the button becomes disabled. Defaults to `type="button"`; pass `type="submit"`
inside the form it saves.

All props from [Button](/components/actions/button), with these differences:

| Prop          | Type        | Default     | Description                                          |
| ------------- | ----------- | ----------- | ---------------------------------------------------- |
| `appearance?` | `string`    | `"filled"`  | The Button appearance, made optional with a default. |
| `intent?`     | `string`    | `"neutral"` | The Button intent, made optional with a default.     |
| `children`    | `ReactNode` | —           | Required — the visible action label is app voice.    |

### Sandbar.DiscardButton

The secondary (discard/reset) action — a mantle [Button](/components/actions/button). Discard is
destructive-ish: for large forms, consider confirming before discarding.

All props from [Button](/components/actions/button), with these differences:

| Prop          | Type        | Default      | Description                                          |
| ------------- | ----------- | ------------ | ---------------------------------------------------- |
| `appearance?` | `string`    | `"outlined"` | The Button appearance, made optional with a default. |
| `intent?`     | `string`    | `"neutral"`  | The Button intent, made optional with a default.     |
| `children`    | `ReactNode` | —            | Required — the visible action label is app voice.    |

### Sandbar.Error

A danger [Alert](/components/feedback/alert) row for a failed save, rendered on its own row of the
panel. Safe to conditionally mount (`{error != null && <Sandbar.Error>{error}</Sandbar.Error>}`):
the announcement goes through Root's persistent assertive announcer, not the mounted node. Needs
Title- or Icon-level control? Compose your own `<Alert.Root intent="danger">` inside
`Sandbar.Root` instead.

All props from [Alert](/components/feedback/alert)'s `Alert.Root` except `intent`, which is pinned
to `"danger"`.

<PageHeader id="app-layout">App Layout</PageHeader>

A viewport-locked application shell. The shell itself never scrolls: `AppLayout.Content` is a rounded card
that scrolls internally with `overscroll-none`, so long pages never bounce or drag the chrome. Pinned to
the top of that card, `AppLayout.Header` is a sticky toolbar row for the sidebar trigger and breadcrumbs,
and `AppLayout.Notice` is a full-window composition slot for app-wide messaging that pushes everything —
including any sidebar — down. It can stack a rare impersonation or environment banner with a persistent
account-alert bar. It is a **layout**: it owns _where things go_, and app state (routing, sessions, data)
never enters it.

The canonical composition pairs the shell with the [Sidebar](/components/navigation/sidebar). The two are
deliberately decoupled — `AppLayout` never references sidebar state — but cohesive: wrap the shell in
`Sidebar.Root`, place `Sidebar.Nav` as the first child of `AppLayout.Body`, and put `Sidebar.Trigger` in
the top-left of `AppLayout.Header`. The
[collapsed icon rail](/components/navigation/sidebar#collapsing-to-the-icon-rail) is entirely the sidebar's
business — the shell markup does not change, and the rail shares the shell's `bg-base` surface, so the card
gutter stays visually seamless. The demo runs in its own framed document, exactly like a real app: collapse
the sidebar to its icon rail with the trigger (or <kbd>⌘B</kbd>/<kbd>Ctrl+B</kbd> — the shortcut binds to
the frame, not this page) and reopen it the same way, toggle the
notice to see the strip push the whole shell (sidebar included) down, and switch the preview to the mobile
viewport for the sheet presentation. The demo narrows the sidebar's `mobileBreakpoint` to `md`
so the desktop panel survives the preview's desktop and tablet widths; ngrok's dashboards keep the `lg`
default.

> [!NOTE]
> The full-shell demos on this page render inside framed previews — each is a separate document, exactly
> like a real app owning the viewport. That is why they can pin `AppLayout.Root` with
> `className="fixed inset-0"` (mantle's `cx` is tailwind-merge-backed, so the override is deterministic) and
> compose a real [Main](/components/primitives/main) landmark — see [Landmarks](#landmarks). When you embed
> the shell inside an existing page instead — as the inline [Polymorphism](#polymorphism) example below
> does — drop both: `AppLayout.Root` fills its nearest sized ancestor by default.

## Single-product bridge

Most ngrok surfaces are still single-product today. This variant keeps that information architecture while
adopting the new shell, so teams can migrate now and layer in product segmentation later. The **account
switcher moves to the top** — where the multi-product shell above puts a product switcher — the body holds a
single product's navigation, and the footer stacks a few pinned links, a `Sidebar.Separator`, and a Help
`DropdownMenu`. When product segmentation lands, put a product switcher in the header and move the account
switcher into the footer; nothing else about the shell changes.

## Composition

```text showLineNumbers=false
AppLayout.Root
├── AppLayout.Notice
└── AppLayout.Body
    └── AppLayout.Inset
        └── AppLayout.Content
            └── AppLayout.Header
```

- **`Root`** is the outer frame (`relative isolate flex h-full w-full flex-col overflow-hidden bg-base`).
  It fills its nearest sized ancestor; merge `className="fixed inset-0"` to pin a real app shell to the
  viewport. The shell never scrolls — only `Content` does.
- **`Notice`** is the full-window-width, vertical composition slot pinned above everything else
  (`flex w-full shrink-0 flex-col`). Stack an impersonation or environment banner and an account alert bar
  here; each brings its own colors while the slot pushes the whole shell down. Collapses to nothing when
  empty. Optional.
- **`Body`** is the main flex row (`flex min-h-0 flex-1`). Its children are the app's columns — place a
  [`Sidebar.Nav`](/components/navigation/sidebar) (or any other rail) before `AppLayout.Inset`. The
  `min-h-0` is what lets the content card scroll instead of the page.
- **`Inset`** is the content column (`flex min-w-0 flex-1 flex-col gap-2 p-2`): it wraps the `Content`
  card and owns the card gutter, keeping a consistent gap beside a sidebar.
- **`Content`** is the scrolling content card:
  `bg-card border-card-muted scrollbar min-h-0 flex-1 overflow-y-auto overscroll-none rounded-xl border shadow-sm`
  — the shell's only scroll container. In a real app shell it should almost always be the `Main` landmark —
  see [Landmarks](#landmarks).
- **`Header`** is the sticky toolbar `<header>` rendered as `Content`'s **first child** (`sticky top-0 h-14`
  with a hairline border): page content scrolls beneath it. The natural home for a `Sidebar.Trigger` in its
  top-left, followed by [Breadcrumb](/components/navigation/breadcrumb) navigation or page actions. Its
  `h-14` plus the 8px card gutter puts it on the same horizontal band as `Sidebar.Header`'s switcher row, so
  the app switcher and content header read as one aligned row. Optional.

## Landmarks

Exactly one element in a document may be the Main landmark, so `AppLayout.Content` renders a plain `<div>`
by default — composing the shell inside a page that already owns the landmark (a docs page, for example) is
safe by default. But when the shell owns the document — which is the normal case for an app shell —
`AppLayout.Content` should almost always **be** the Main landmark: compose mantle's
[Main](/components/primitives/main) via `asChild`, and pair a
[SkipToMainLink](/components/primitives/skip-to-main-link) as the first child of `AppLayout.Root`. Its
default `targetId` matches `Main`'s `id="main"`, so keyboard users can jump past the sidebar's tab gauntlet
and the toolbar straight into the content card. The framed demos on this page do exactly this — each preview
is its own document, so each owns a real `Main`.

```tsx
import { AppLayout } from "@ngrok/mantle/app-layout";
import { Main } from "@ngrok/mantle/main";
import { SkipToMainLink } from "@ngrok/mantle/skip-to-main-link";

<AppLayout.Root className="fixed inset-0">
	<SkipToMainLink />
	<AppLayout.Body>
		<AppLayout.Inset>
			<AppLayout.Content asChild>
				<Main>{/* your routed page content */}</Main>
			</AppLayout.Content>
		</AppLayout.Inset>
	</AppLayout.Body>
</AppLayout.Root>;
```

## Polymorphism

Every part accepts an `asChild` prop. When `true`, the part renders its single child instead of its default
element, forwarding all class names, `data-*` attributes, and the ref onto that child.

## Accessibility

- `AppLayout.Header` renders a semantic `<header>` scoped to the content region — inside a `Main`-composed
  `AppLayout.Content` it is deliberately not an ARIA `banner` landmark.
- `AppLayout.Notice` is deliberately **not** named Banner — it renders a plain `div` and claims no landmark.
  Give the notice content its own semantics if it needs them.
- Compose `Main` into `AppLayout.Content` (via `asChild`) and render `SkipToMainLink` first in the shell so
  keyboard users can jump over the sidebar and toolbar into the content card.
- The shell never scrolls; only `AppLayout.Content` does. Focus scrolling behaves like any regular scroll
  container.

## API Reference

Every part accepts a `ref` prop that lands on the rendered element and merges `className` deterministically
via `cx` (tailwind-merge-backed, so consumer classes win).

### AppLayout.Root

The outer frame of the shell: `relative isolate flex h-full w-full flex-col overflow-hidden bg-base`. Fills
its nearest sized ancestor; merge `className="fixed inset-0"` to pin a real app shell to the viewport. The
shell itself never scrolls — only `AppLayout.Content` does. When the shell owns the document, render a
[SkipToMainLink](/components/primitives/skip-to-main-link) as its first child (see
[Landmarks](#landmarks)).

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                                                                                           |
| ---------- | --------- | ------- | --------------------------------------------------------------------------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Use the `asChild` prop to compose the `AppLayout.Root` styling onto alternative element types or your own components. |

### AppLayout.Notice

A full-window-width, vertical composition slot pinned above everything else in the shell — including any
sidebar — for app-wide messaging. Stack a rare impersonation or environment banner and a persistent account
alert bar here; each child brings its own colors and layout while the slot pushes the whole shell down.
Renders an unstyled `<div>` (`flex w-full shrink-0 flex-col`) and collapses to nothing when empty.
Deliberately not named Banner — `AppLayout.Header` renders the page's `<header>` (the ARIA
`banner` landmark), and this part is not that. Optional.

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                                                                                             |
| ---------- | --------- | ------- | ----------------------------------------------------------------------------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Use the `asChild` prop to compose the `AppLayout.Notice` styling onto alternative element types or your own components. |

### AppLayout.Body

The main horizontal region below the notice: a flex row (`flex min-h-0 flex-1`) whose children are the app's
columns — place a [`Sidebar.Nav`](/components/navigation/sidebar) (or any other rail) before
`AppLayout.Inset`. The `min-h-0` is what lets `AppLayout.Content` scroll instead of the page.

### AppLayout.Inset

The content column beside the sidebar: `flex min-w-0 flex-1 flex-col gap-2 p-2`. Wraps the
`AppLayout.Content` card and owns the card gutter, keeping a consistent gap beside a sidebar (invisible when
both share `bg-base`).

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                                                                                            |
| ---------- | --------- | ------- | ---------------------------------------------------------------------------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Use the `asChild` prop to compose the `AppLayout.Inset` styling onto alternative element types or your own components. |

### AppLayout.Header

The sticky toolbar row at the top of the content card — render it as `AppLayout.Content`'s **first child**
(`sticky top-0 z-10 h-14` with a `border-card-muted` hairline beneath it, so page content scrolls under it).
The natural home for a [`Sidebar.Trigger`](/components/navigation/sidebar#sidebartrigger) in its top-left,
followed by [Breadcrumb](/components/navigation/breadcrumb) navigation, search, or page-level actions. Its
`h-14` plus the 8px card gutter vertically aligns it with `Sidebar.Header`'s switcher row. Renders a
semantic `<header>` scoped to the content region (inside `Main` it is not an ARIA `banner` landmark).
Optional.

All props from [header](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/header#attributes), plus:

| Prop       | Type      | Default | Description                                                                                                             |
| ---------- | --------- | ------- | ----------------------------------------------------------------------------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Use the `asChild` prop to compose the `AppLayout.Header` styling onto alternative element types or your own components. |

### AppLayout.Content

The scrolling content card of the shell:
`bg-card border-card-muted scrollbar min-h-0 flex-1 overflow-y-auto overscroll-none rounded-xl border shadow-sm`.
The shell's only scroll container — the page body never scrolls, and `overscroll-none` keeps scroll from
bouncing the chrome. Render `AppLayout.Header` as its first child for a sticky in-card toolbar. The card
renders a `<div>` by default, but in a real app shell this should almost always be the
[Main](/components/primitives/main) landmark composed via `asChild` (see [Landmarks](#landmarks)).

All props from [div](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/div#attributes), plus:

| Prop       | Type      | Default | Description                                                                                                                                           |
| ---------- | --------- | ------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `asChild?` | `boolean` | `false` | Use the `asChild` prop to compose the `AppLayout.Content` styling onto alternative element types (e.g. `Main`, a `<section>`) or your own components. |
